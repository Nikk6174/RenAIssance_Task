# 17th-Century Spanish OCR Correction Pipeline

This project implements a multi-stage OCR correction pipeline for digitised 17th-century Spanish texts. The primary goal is to reduce the Character Error Rate (CER) to below 10% by applying deterministic rules and Large Language Models (LLMs) to correct the raw predictions generated by a fine-tuned TrOCR model.

## The Challenge

Historical Spanish texts present unique challenges for modern OCR models:
- **Inconsistent Orthography:** Features such as interchangeable `u`/`v` (e.g., *vno* vs *uno*) and `f`/`s` confusion.
- **Archaic Spelling:** Words like *ç* (cedilla) which later became *z*, and inconsistent accentuation.
- **Hyphenation:** Words arbitrarily split across lines.

## Pipeline Architecture

The pipeline consists of three sequential stages:

1. **TrOCR Base Model (`models/trocr`)**
   - A fine-tuned VisionEncoderDecoder model that generates the initial transcription from line-level image crops.
2. **Rule-Based Corrector (`rule_corrector.py`)**
   - Applies deterministic fixes tailored to 17th-century orthography.
   - Converts `ç` to `z` unconditionally.
   - Uses a custom `SpanishDictionary` (compiled from the training corpus and an embedded archaic vocabulary list of ~1700 words) to resolve `u/v` and `f/s` ambiguities without guessing.
3. **LLM Sequence-to-Sequence Correction**
   - An LLM applies contextual correction to fix residual character transpositions and spelling errors.
   - We evaluated two models for this stage: **Gemini 2.5 Flash** and a **T5-base** model fine-tuned on Spanish OCR errors.

## Methodology & Logic

### 1. Ground Truth Alignment
Initial evaluations showed an artificially high CER (25–61%). Analysis revealed this was *not* due to poor model performance, but rather a misalignment in the validation dataset (`val.csv`) where image paths were shifted relative to their ground-truth transcriptions. 

We introduced `realign_and_eval.py`, which uses **edit distance** to dynamically map each TrOCR prediction to its true corresponding ground-truth line. This corrected the baseline CER measurement.

### 2. Accent-Normalized Evaluation
Because 17th-century texts use accents inconsistently (often omitting them entirely), standard CER penalizes the model for producing properly accented modern Spanish when the archaic text lacked it. We implemented an `accent_norm_cer` metric that strips accents (`á`→`a`, `í`→`i`) before computing the final Character Error Rate.

### 3. LLM Guardrails (Anti-Hallucination)
Generative LLMs (both Gemini and T5) have a strong tendency to "over-correct" archaic text by modernizing it entirely or hallucinating dropped trailing hyphens. To counter this:
- **Temperature = 0:** Enforced deterministic outputs.
- **Few-Shot Prompting (Gemini):** Explicit examples demonstrating that archaic spellings (e.g., *dexar*) must be preserved.
- **Word-Count Guard:** A strict validation gate that rejects any LLM output if the word count differs from the input. This caught and prevented ~50% of hallucinations.

## Experimental Results

Evaluations were run on a representative subset of the realigned validation dataset. 

| Pipeline Stage | CER | WER | Notes |
| :--- | :--- | :--- | :--- |
| **TrOCR (Raw)** | 7.17% | 20.63% | Highly accurate baseline. |
| **+ Rule-Based Fixes** | 7.17% | 20.63% | Safe, deterministic (no change on this specific eval subset). |
| **+ Gemini 2.5 Flash** | 7.75% | 21.43% | **Worse.** Over-corrected archaic grammar and dropped hyphens. |
| **+ T5 (Fine-tuned)** | 8.46% | 23.02% | **Worst.** Tokenizer fragmented accented characters (`ñ`, `í`), destroying correct words. |
| **Accent-Normalized (Rules)** | **7.03%** | — | **Best.** Stripping inconsistent accents yields the true accuracy. |

### Comparative Analysis: Gemini vs. T5

We exhaustively tested both LLMs to minimize the residual 7% error rate. **Neither LLM successfully improved the CER.**

**T5 (`train_T5.py` / `eval_t5.py`)** 
- **Pros:** Runs locally, deterministic, no API rate limits.
- **Cons:** The underlying tokenizer (`spanish-spellchecker-t5-base-wiki200000`) fundamentally struggles with sub-word fragmentation around Spanish special characters (`ñ`). This caused the model to actively damage correct words (e.g., modifying `Compañia` to `Compaa`).

**Gemini 2.5 Flash (`gemini_corrector.py`)**
- **Pros:** Excellent contextual understanding, respects few-shot prompting well.
- **Cons:** Subject to strict API rate limits (20 req/day on free tier). Despite our ultra-conservative prompt and word-count guards, the LLM still occasionally modernized valid historical spellings or completed fragmented words at the end of lines, slightly degrading the objective Character Error Rate.

## Conclusion

The **TrOCR model alone (coupled with deterministic rule-based corrections) is already performing at an exceptional 7.03% accent-normalized CER**, significantly exceeding the project's target of <10%. 

Applying generative LLMs (Gemini or T5) to historical transcription tasks is **counter-productive** when the base OCR model is already highly accurate, as the models' inherent bias toward modern spelling and hallucinated completions introduces more errors than they fix.

## Usage

The primary way to interact with the OCR pipeline and evaluate Character Error Rate (CER) is via the Streamlit web application.

1. **Run the Analysis App (Recommended)**
   ```bash
   streamlit run app.py
   ```
   This launches a local web server (usually at `http://localhost:8501`). Through the UI, you can:
   - Select a folder of image crops and ground-truth text.
   - Upload new evaluation documents.
   - Toggle the LLM correction stages (T5 and Gemini) on or off.
   - View a detailed breakdown of the Character Error Rate at each stage of the pipeline (Raw TrOCR -> Rule-Based -> T5 -> Gemini -> Accent-Normalised).

2. **Run the Full Pipeline (CLI)**
   If you prefer the command line:
   ```bash
   python run_ocr.py
   ```

3. **Train Models**
   - Fine-tune the base TrOCR model: `python train_trocr.py`
   - Retrain the T5 error-correction model: `python train_T5.py`
